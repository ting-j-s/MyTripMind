## 1. 项目概述

### 1.1 项目名称
个性化旅游系统（Personalized Travel System）

### 1.2 项目类型
前后端分离的Web应用

### 1.3 系统架构

```
┌─────────────────────────────────────────────────────────┐
│                      用户浏览器                          │
│                    （HTML/CSS/JS）                       │
└─────────────────────────┬───────────────────────────────┘
                          │ HTTP请求
                          ▼
┌─────────────────────────────────────────────────────────┐
│                   Python Flask 后端                      │

└─────────────────────────┬───────────────────────────────┘
                          │
                          ▼
```

| 层级 | 技术选择 | 说明 |
|------|----------|------|

## 4. 功能模块详细需求

### 4.1 模块一：旅游推荐

**功能清单**：

| 功能 | 说明 | 前端 | 后端API |
|------|------|------|---------|
| 景点列表 | 展示所有景点 | GET /api/attractions | 返回景点列表 |
| 景点搜索 | 按名称、类别搜索 | GET /api/attractions/search?q=关键词 | 模糊搜索 |
| 景点排序 | 按热度/评分排序 | GET /api/attractions?sort=heat | Top-K排序 |
| 个性化推荐 | 根据用户兴趣推荐 | GET /api/recommend | 推荐算法 |

**数据结构**：
```json
{
  "id": "ATTR_001",
  "name": "故宫",
  "type": "景区",
```

## 5. JSON 数据文件设计

### 5.1 数据文件

本项目使用 JSON 文件存储全部数据，位于 `backend/data/` 目录下，
通过 loader/storage/model/service 模块读写，不依赖数据库。

| 文件 | 存储内容 | 顶层结构 |
|------|---------|---------|
| users.json | 用户信息（用户名、密码、兴趣、收藏） | `{"users": [...]}` |
| attractions.json | 景点信息（名称、类型、坐标、热度、评分、标签） | `{"attractions": [...]}` |
| buildings.json | 建筑信息（名称、坐标、楼层、所属景点） | `{"buildings": [...]}` |
| facilities.json | 服务设施（名称、类型、坐标、所属建筑） | `{"facilities": [...]}` |
| roads.json | 道路网络（起点、终点、距离、速度、拥挤度、交通方式） | `{"roads": [...]}` |
| diaries.json | 用户日记（标题、内容、关联景点、评分、压缩内容） | `{"diaries": [...]}` |
| foods.json | 美食信息（名称、菜系、餐厅、坐标、热度、评分） | `{"foods": [...]}` |

### 5.2 数据读写流程

```
前端 HTTP API → routes/ → services/ → models/ → loader/storage → backend/data/*.json
```

## 12. 关键技术细节（容易被忽略）

### 12.1 TSP问题是闭环的

> "从当前位置出发，参观多个景点后返回当前位置"

```
不是：Dijkstra(A→B) + Dijkstra(B→C)
而是：TSP环游问题（起点=终点）
```

**算法**：贪心最近邻 + 2-opt优化

### 12.2 交通工具约束

| 交通方式 | 可走路类型 |
|----------|------------|
| 步行 | 全部道路 |
| 自行车 | 仅 `road_types` 包含"自行车"的道路 |
| 电瓶车 | 仅 `road_types` 包含"电瓶车"的道路 |

**注意**：混合模式时，同一段路只能选一种交通方式

### 12.3 拥挤度与时间计算

```python
真实速度 = 拥挤度 × 理想速度
# 拥挤度范围：(0, 1]，越大越堵
# 时间 = 距离 / 真实速度
```

### 12.4 室内导航三级结构

```
Level 1: 大门 → 电梯口（楼内平面）
Level 2: 电梯（楼层间移动）
Level 3: 电梯口 → 目标房间（楼层内平面）
```

**实现**：室内也用图结构，电梯作为特殊节点连接不同楼层

### 12.5 场所查询必须用路径距离

```
❌ 错误：直线距离排序
✅ 正确：基于道路网络计算的实际最短路径距离
```

### 12.6 数据动态变化

```
问题：用户浏览/收藏后，热度/评分变化，Top-10要重新计算
解决：不用每次全量排序，用"增量更新"机制
- 每次数据变化，只更新受影响的元素
- 维护一个"候选池"，减少排序范围
```

### 12.7 最短路径算法变体

| 场景 | 算法 |
|------|------|
| 单源最短路径 | Dijkstra |
| 单源最短时间（考虑拥挤度） | Dijkstra（边权重=时间） |
| 交通工具约束 | Dijkstra（过滤边） |
| 室内导航 | Dijkstra（特殊图结构） |
