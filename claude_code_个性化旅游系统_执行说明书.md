# 个性化旅游系统（数据结构课程设计）Claude Code 执行说明书

## 1. 任务目标
请你为我完成一个**“个性化旅游系统”数据结构课程设计项目**。  
本项目的核心不是做一个花哨的 Web 页面，也不是把功能堆到数据库里，而是：

1. **围绕真实业务场景设计并实现数据结构与算法**
2. **所有要求的功能都必须可运行、可演示**
3. **核心算法必须基于你自己设计的数据结构，并自行编程实现**
4. **不要使用数据库来完成核心功能**
5. **不要用“完全排序”“直线距离”“现成黑盒库算法”去糊弄核心考核点**

系统架构和开发语言不限，但最终必须是一个**真实可用、可运行、可演示**的系统。

---

## 2. 你必须严格遵守的硬性约束（非常重要）

### 2.1 课程考核重点
本课程设计**考核重点不是 UI / Web 开发 / 数据库**，而是：
- 常用数据结构与算法在实际工程中的实现
- 针对具体场景选择合适的数据结构与算法
- 对时间复杂度 / 空间复杂度进行权衡
- 尽量自己“造轮子”，体现数据结构课程的训练价值

### 2.2 禁止事项
1. **不要使用数据库完成核心功能**
   - 核心功能指：推荐、查询、排序、路径规划、全文检索、压缩存储、附近设施检索、美食 Top-K、室内导航等。
   - 这些核心能力必须由**自定义数据结构 + 自实现算法**完成。
2. **不要把“前10推荐”通过先全量排序再截取前10来实现**
   - 必须用**部分排序 / Top-K**思路实现。
3. **不要把附近设施排序建立在“直线距离”上**
   - 必须基于**道路图/可通行路径的实际距离或实际代价**。
4. **不要只做静态展示**
   - 所有模块都要可交互、可查询、可演示。
5. **不要照搬往年报告中的冲突实现**
   - 往年报告里出现过 SQLite / Redis / 第三方 AI 模块等内容，那些只能当展示思路参考，不能违背本学期“核心功能不能依赖数据库”的要求。

### 2.3 允许事项
- 可以做 Web、桌面端或命令行 + 可视化页面
- 可以使用文件进行数据持久化（如 JSON / CSV / 自定义二进制文件）
- 可以为非核心功能适度使用框架，但**核心算法必须自己写**

## 3. 本项目必须完整实现的功能范围

你必须完成以下**全部要求**，不能漏模块。

### 3.1 旅游推荐
必须支持：
1. 用户根据喜好选择景点和学校作为旅游目的地
2. 系统在旅游前推荐景点和学校
3. 推荐维度至少包括：
   - 热度
   - 评价
   - 个人兴趣
4. 用户可按名称、类别、关键字进行查询
5. 查询结果有多项时，可按热度和评价排序

#### 核心算法要求
- 推荐基础要求是**排序算法**
- 因为用户通常只看前10个结果，必须实现：
  - **不经过完全排序即可得到前10个景点/学校**
  - 并且要考虑**数据动态变化**
- 查询部分必须体现：
  - **查找算法**
  - **排序算法**

#### 推荐模块建议实现方向
你应优先采用如下组合：
- 存储：数组 / 动态数组 + 哈希索引 + 倒排索引
- Top-K：小顶堆 / 优先队列
- 关键字查询：倒排索引、Trie、哈希表等
- 个人兴趣：标签向量、权重映射、自定义评分函数

---

### 3.2 旅游路线规划
必须支持：
1. 用户输入目标景点/场所，规划从当前位置出发到目标点的最优路线
2. 用户输入多个目标景点/场所，规划从当前位置出发、参观多个点、并**返回当前位置**的最优路线
3. 需要有图形化导航界面：
   - 地图展示
   - 路径展示

#### 路线策略必须包含
1. **最短距离策略**
2. **最短时间策略**
   - 每条道路有拥挤度
   - 拥挤度为 `<= 1` 的正数
   - 真实速度 = 拥挤度 × 理想速度
   - 每条道路的拥挤度与理想速度可自拟
3. **交通工具最短时间策略**
   - 校区：步行 / 自行车
   - 景区：步行 / 电瓶车
   - 自行车只能走自行车道路
   - 电瓶车只能走固定电瓶车路线
   - 必须考虑不同拥挤度下的最短时间
   - **允许混合交通工具**
4. **室内导航策略**
   - 需要模拟教学楼 / 博物馆等建筑内部结构
   - 至少包括：
     - 大门到电梯导航
     - 楼层间电梯导航
     - 楼层内到房间导航

#### 核心算法要求
- 单目标：**最短路径算法**
- 多目标：**途经多点最短路径算法**
- 室内导航：仍应基于图搜索 / 网格搜索 / 分层图搜索自行实现

#### 路线模块建议实现方向
- 图存储：邻接表
- 单目标最短路：Dijkstra / A*（若使用 A*，启发函数需自行设计）
- 多目标路径：状态压缩 DP / 近似 TSP / 分阶段最短路拼接（要解释选择理由）
- 室内导航：楼层图 + 电梯/楼梯跨层边 + 栅格图 / 可达图

---

### 3.3 场所查询
在景区或学校内部，必须支持：
1. 选中某个景点/场所后，找出附近一定范围内的超市、卫生间等设施
2. 按距离排序
3. 按类别过滤
4. 用户输入类别名称后查找某地点附近的对应服务设施
5. 查询结果按距离排序

#### 核心算法要求
- 排序
- 查找
- **距离不能使用直线距离**
- 必须基于道路网络上的**实际可达距离/路径代价**

#### 场所查询建议实现方向
- 空间索引候选：网格索引 / GeoHash / 四叉树
- 候选过滤后，再走道路图最短路求真实距离
- 排序：堆 / 快排 / 归并 / 多关键字排序

---

### 3.4 旅游日记管理
必须支持：
1. 用户在旅游中或旅游后撰写旅游日记
2. 日记支持：
   - 文字
   - 图片
   - 视频
3. 对所有用户的旅游日记进行统一管理
4. 用户可以浏览和查询所有用户的旅游日记
5. 浏览量作为日记热度
6. 用户浏览后可以对日记评分
7. 浏览所有日记时，可以按照：
   - 热度
   - 评价
   - 个人兴趣
   进行推荐

#### 核心算法要求
- 推荐基础为**排序算法**
- 热度、评分推荐必须体现排序算法设计
- 个人兴趣推荐要有明确的数据结构和评分逻辑

#### 日记管理建议实现方向
- 日记主存储：顺序表 / 链表 / 文件索引
- 按热度/评分动态维护：平衡树 / 堆 / 跳表 / 多索引结构
- 用户兴趣：标签哈希表 + 分值模型

---

### 3.5 旅游日记交流
必须支持：
1. 用户输入旅游目的地，对相关日记按热度和评分排序
2. 用户输入日记名称进行**精确查询**
3. 可按日记内容进行**全文检索**
4. 日记支持**压缩存储**
5. 使用 AIGC 算法，根据照片、文字描述等信息生成旅游动画

#### 核心算法要求
- 目的地相关日记：查找 + 排序
- 日记名称精确查询：
  - 需考虑“日记数量较大、变化很快”的情况
  - 要实现**高效查找**
- 内容全文检索：
  - 必须体现**文本搜索算法**
- 压缩存储：
  - 必须体现**无损压缩算法**

#### 日记交流建议实现方向
- 名称精确查询：哈希表 / 平衡树 / Trie
- 全文检索：倒排索引 / Trie + 分词 / KMP（用于子串检索场景）
- 压缩：Huffman / LZ77 / LZW / 自定义组合无损压缩
- AIGC 动画：可对接现成模型或接口，但请把它视为功能项，不要把它当作本课程的数据结构考核核心

---

### 3.6 美食推荐
在选中游览景点和学校后，必须支持：
1. 按热度、评价、距离排序
2. 按菜系过滤
3. 输入美食名称、菜系、饭店或窗口名称进行基于内容的模糊查询
4. 查询结果多项时，可按热度、评价、距离排序

#### 核心算法要求
- 排序算法
- 模糊查找算法
- 因为用户通常只看前10个美食：
  - 必须实现**不经过完全排序即可得到前10个美食**

#### 美食推荐建议实现方向
- 倒排索引 / Trie / n-gram 实现模糊查找
- 小顶堆 / 优先队列实现 Top-K
- 距离仍建议基于道路图或场所图上的实际代价
