## 1. 项目概述

### 1.1 项目背景

传统旅游系统功能单一，缺乏个性化推荐和智能化路线规划。本系统旨在为用户提供一个集景点推荐、路线规划、场所查询、旅游日记分享和美食推荐于一体的个性化旅游管理平台。

### 1.2 系统目标

- 实现基于热度、评分、兴趣的个性化景点推荐
- 提供多种策略的最短路径和TSP多景点路线规划
- 支持基于道路网络的附近设施查询
- 实现旅游日记的创建、分享、评分和全文检索
- 提供美食推荐和模糊搜索功能

### 1.3 技术约束

1. **核心算法必须自己实现**，禁止直接调用数据库的排序、查找
2. **必须使用自己设计的数据结构**（堆、哈希表、链表、图等）
3. **不使用数据库存储核心功能**，采用JSON文件存储
4. 考核重点：**数据结构的设计及选择、算法复杂度平衡**

## 3. 数据结构设计

### 3.1 核心数据结构选型

| 数据结构 | 选用理由 | 应用场景 |
|----------|----------|----------|
| **图（邻接表）** | 道路网络是稀疏图，邻接表空间效率高 | 存储景点/设施间的道路连接 |
| **堆（最小堆）** | Dijkstra需要快速获取最短距离节点 | 优先队列、Top-K排序 |
| **哈希表** | O(1)查找比遍历O(n)快很多 | 用户ID、日记ID快速查询 |
| **链表** | 动态扩展，适合日记列表 | 日记、美食等列表存储 |

## 4. 核心算法方案

### 4.1 最短路径算法 - Dijkstra

**算法选择**：Dijkstra + 最小堆

| 变体 | 边权重 | 应用场景 |
|------|--------|----------|
| 标准Dijkstra | distance | 最短距离策略 |
| 时间Dijkstra | distance/(congestion×speed) | 最短时间策略 |
| 约束Dijkstra | 边过滤 | 交通工具约束 |

**复杂度**：
- 时间：O((V+E) log V)
- 空间：O(V)

### 4.2 多景点路线 - TSP

**算法选择**：贪心最近邻 + 2-opt优化

```
贪心最近邻：O(n²) - 快速得到初始解
    ↓
2-opt优化：O(n²×iter) - 迭代改进解质量
```

**为什么不用精确算法**：
- TSP是NP难问题，精确算法无法在合理时间内求解
- 贪心+局部搜索是工程上常用的近似算法

### 4.3 Top-K排序

**算法选择**：维护大小为k的小顶堆

```
时间复杂度：O(n log k)
空间复杂度：O(k)
适用场景：只取前10个，不需要完全排序
```

### 4.4 搜索算法

| 算法 | 方案 | 复杂度 |
|------|------|--------|
| 模糊搜索 | 编辑距离（Levenshtein） | O(n×m×k) |
| 全文检索 | 倒排索引 | O(1)查找 + O(n)返回 |

### 4.5 压缩算法

**算法选择**：霍夫曼编码

- 变长编码，高频字符用短码
- 无损压缩
- 压缩率：30%-50%

## 9. 关键技术决策

### 9.1 Dijkstra为什么用堆优化？

**问题**：普通Dijkstra需要O(V²)找到最小距离节点

**解决方案**：使用最小堆维护待处理节点
- 堆顶总是当前最短距离节点
- 更新邻居距离时，push到堆中
- 最终复杂度：O((V+E) log V)

### 9.2 Top-K为什么不直接排序？

**问题**：n个景点，用户只看前10个

**方案对比**：
| 方案 | 时间复杂度 | 200个景点耗时 |
|------|-----------|--------------|
| 全排序 | O(n log n) | O(200×8) = 1600 |
| Top-K堆 | O(n log k) | O(200×4) = 800 |

### 9.3 哈希表为什么自己实现？

**课程要求**：核心功能不能使用数据库

**方案**：
- 使用链地址法（拉链法）解决哈希冲突
- 支持动态扩容
- O(1)平均查找用户ID、日记ID

### 9.4 为什么不使用数据库？

课程明确要求：**不要使用数据库来完成核心功能**

**替代方案**：
- JSON文件存储所有数据
- 自己实现HashTable管理用户/日记ID
- 自己实现排序/搜索算法
